## 1. 機会

### 発見の経緯

複数のソースにわたるウェブリサーチにより、3つの収束するトレンドが特定されました：

- **AIインフラ/オーケストレーション**は最も成長の速いソフトウェアカテゴリです（AIオーケストレーション市場は2026年に約135億ドル、CAGR 22%超）
- **空間コンピューティング**（Vision Pro、WebXR）は成熟しつつありますが、キラーとなるエンタープライズアプリが不足しています
- 既存のAIワークフローツール（LangSmith、n8n、Flowise、CrewAI）はすべて**フラットな2Dダッシュボード**です

### コンセプト：Nexus Spatial

空間コンピューティングにおけるAIエージェントのコマンドセンター ── VisionOS + WebXR アプリケーションで、AIエージェントのオーケストレーション、モニタリング、インタラクションのための没入型3Dコマンドセンターを提供します。ユーザーはエージェントパイプラインを3Dノードグラフとして可視化し、空間パネルでリアルタイムの出力を監視し、3D空間でのドラッグ&ドロップによりワークフローを構築し、共有空間環境でコラボレーションします。

### このエージェンシーが独自のポジションを持つ理由

エージェンシーは空間コンピューティングの深い専門知識（XR開発者、VisionOSエンジニア、Metalスペシャリスト、インターフェースアーキテクト）と、フルスタックのエンジニアリング・デザイン・マーケティング・オペレーションを兼ね備えています ── 空間コンピューティングの習熟とエンタープライズソフトウェアの厳密さの両方を要求する製品には稀な組み合わせです。

## 3. 技術アーキテクチャ

**エージェント：** Backend Architect

### システム概要

水平スケーリングとプロバイダーに依存しないAI統合のために設計された、明確な所有権境界を持つ8サービスアーキテクチャ。

```
+------------------------------------------------------------------+
|                     CLIENT TIER                                   |
|  VisionOS Native (Swift/RealityKit)  |  WebXR (React Three Fiber) |
+------------------------------------------------------------------+
                              |
+-----------------------------v------------------------------------+
|                      API GATEWAY (Kong / AWS API GW)              |
|  Rate limiting | JWT validation | WebSocket upgrade | TLS        |
+------------------------------------------------------------------+
                              |
+------------------------------------------------------------------+
|                      SERVICE TIER                                 |
|  Auth | Workspace | Workflow | Orchestration (Rust) |             |
|  Collaboration (Yjs CRDT) | Streaming (WS) | Plugin | Billing    |
+------------------------------------------------------------------+
                              |
+------------------------------------------------------------------+
|                      DATA TIER                                    |
|  PostgreSQL 16 | Redis 7 Cluster | S3 | ClickHouse | NATS        |
+------------------------------------------------------------------+
                              |
+------------------------------------------------------------------+
|                    AI PROVIDER TIER                                |
|  OpenAI | Anthropic | Google | Local Models | Custom Plugins      |
+------------------------------------------------------------------+
```

### 技術スタック

| コンポーネント | 技術 | 選択理由 |
|-----------|------------|-----------|
| オーケストレーションエンジン | **Rust** | サブミリ秒スケジューリング、GCポーズなし、エージェントサンドボックスのメモリ安全性 |
| APIサービス | TypeScript / NestJS | CRUDが多いサービスの開発速度向上 |
| VisionOS クライアント | Swift 6, SwiftUI, RealityKit | Liquid Glass を使ったファーストクラスの空間コンピューティング |
| WebXR クライアント | TypeScript, React Three Fiber | React コンポーネントモデルを使ったプロダクショングレードの WebXR |
| メッセージブローカー | NATS JetStream | 軽量、厳密に一度の配信、Kafka より簡潔 |
| コラボレーション | Yjs (CRDT) + WebRTC | 競合のない並行3Dグラフ編集 |
| プライマリデータベース | PostgreSQL 16 | 柔軟な設定のための JSONB、テナント分離のための行レベルセキュリティ |

### コアデータモデル

14テーブル：
- **ID & アクセス：** users, workspaces, team_memberships, api_keys
- **ワークフロー：** workflows, workflow_versions, nodes, edges
- **実行：** executions, execution_steps, step_output_chunks
- **コラボレーション：** collaboration_sessions, session_participants
- **認証情報：** provider_credentials（AES-256-GCM 暗号化）
- **課金：** subscriptions, usage_records
- **監査：** audit_log（追記専用）

### ノードタイプレジストリ

```
Built-in Node Types:
  ai_agent          -- Calls an AI provider with a prompt
  prompt_template   -- Renders a template with variables
  conditional       -- Routes based on expression
  transform         -- Sandboxed code snippet (JS/Python)
  input / output    -- Workflow entry/exit points
  human_review      -- Pauses for human approval
  loop              -- Repeats subgraph
  parallel_split    -- Fans out to branches
  parallel_join     -- Waits for branches
  webhook_trigger   -- External HTTP trigger
  delay             -- Timed pause
```

### WebSocket チャンネル

WSS によるリアルタイムストリーミング：
- チャンネルごとのシーケンス番号による順序管理
- 再生リクエストによるギャップ検出
- 1000イベント以上遅れた場合のスナップショット回復
- 低性能デバイス向けのクライアントサイドスロットリング

### セキュリティアーキテクチャ

| レイヤー | メカニズム |
|-------|-----------|
| ユーザー認証 | OAuth 2.0（GitHub、Google、Apple）+ メール/パスワード + オプションの TOTP MFA |
| APIキー | SHA-256 ハッシュ化、スコープ制限、任意の有効期限 |
| サービス間通信 | サービスメッシュ経由の mTLS |
| WebSocket 認証 | 30秒有効期限付きのワンタイムチケット |
| 認証情報ストレージ | エンベロープ暗号化（AES-256-GCM + AWS KMS） |
| コードサンドボックス化 | gVisor/Firecracker マイクロ VM（ネットワークなし、RAM 256MB、CPU 30秒） |
| テナント分離 | PostgreSQL 行レベルセキュリティ + S3 IAM ポリシー + NATS サブジェクトスコープ |

**エージェント：** Brand Guardian

### ポジショニング

**カテゴリ競争よりカテゴリ創造。** Nexus Spatial は、過密なAI可観測性ダッシュボード空間での競争ではなく、新カテゴリ ── **空間AIオペレーション（SpatialAIOps）** ── を定義します。

**ポジショニングステートメント：** 複雑なAIエージェントワークフローを管理する技術チームに向けて、Nexus Spatial は、フラットな2Dダッシュボードとは異なり、エージェントオーケストレーションの空間認識を提供する没入型3Dコマンドセンターです。空間コンピューティングはモニタリングをダッシュボードを読む行為からインフラに住む体験へと変革するからです。

### 名称の検証

「Nexus Spatial」は**強力と検証済み：**
- 「Nexus」はNEXUSオーケストレーションフレームワーク（Network of EXperts, Unified in Strategy）に繋がる
- 「Nexus」は独立して「中央の接続点」を意味し ── コマンドセンターに完璧
- 「Spatial」は Apple と業界が標準化した業界標準の表現
- 音節のバランス：3音節と2音節
- **要対応：** ナイス分類第9、42、38類での商標クリアランス

### 主要な発見：デバッグがキラーユースケース

ワークフロー構造上のランタイムトレースの空間オーバーレイは、2Dツールが適切に処理できない実際の定量化されたペインポイントを解決します。このワークフローに最も多くの設計・エンジニアリング投資を行うべきです。

### 重要なデザインインサイト

空間は**構造的な**タスク（ノードの配置、接続、並べ替え）に価値を加えますが、**パラメーター**タスク（テスト入力、設定）には摩擦を生みます。インターフェースは空間と2Dモードをシームレスに組み合わせなければなりません ── 2Dパネルが空間位置にアンカーされる形で。

### 7つのデザイン原則

1. **空間はその価値を証明しなければならない** ── 2Dの方が明確であれば2Dを使う。すべてのレビューで「これはフラットな方が良いか？」と問う
2. **一目でわかる情報、次に詳細** ── 重要情報は色・サイズ・動き・位置により2秒以内に知覚可能
3. **ハンズフリーがベースライン** ── 視線+音声がすべての読み取り/ナビゲーション操作をカバー；手は精度を高めるが必須ではない
4. **認知的重力を尊重する** ── 2Dのメンタルモデル（左から右へのフロー）を拡張し、置き換えない；z軸はレイヤリングを追加
5. **段階的な空間複雑性** ── 新規ユーザーはほぼ2Dから始まる；空間機能は自信が育つにつれて明らかになる
6. **物理的なメタファー、デジタルの能力** ── ノードは「拾い上げる」（物理的）が、複製とバージョン管理も可能（デジタル）
7. **静寂は機能である** ── 正常なシステムは穏やかに感じる；色と動きが通常からの逸脱を示す

## 9. 空間インターフェースアーキテクチャ

### コマンドシアター

ワークスペースはユーザーを囲む湾曲したシアターとして構成されます：

```
                        OVERVIEW CANOPY
                     (pipeline topology)
                    ~~~~~~~~~~~~~~~~~~~~~~~~
                   /                        \
                  /     FOCUS ARC (120 deg)   \
                 /    primary node graph work   \
                /________________________________\
               |                                  |
    LEFT       |        USER POSITION             |       RIGHT
    UTILITY    |        (origin 0,0,0)            |       UTILITY
    RAIL       |                                  |       RAIL
               |__________________________________|
                \                                /
                 \      SHELF (below sightline) /
                  \   agent status, quick tools/
                   \_________________________ /
```

- **フォーカスアーク**（120度、1.2〜2.0m）：プライマリノードグラフのワークスペース
- **オーバービューキャノピー**（上方、2.5〜4.0m）：ミニチュアパイプライントポロジー + 健全性ヒートマップ
- **ユーティリティレール**（左右のフランク）：エージェントライブラリ、モニタリング、ログ
- **シェルフ**（視線より下、0.8〜1.0m）：実行/停止、元に戻す/やり直す、クイックツール

### 3レイヤー奥行きシステム

| レイヤー | 奥行き | コンテンツ | 不透明度 |
|-------|-------|---------|---------|
| フォアグラウンド | 0.8〜1.2m | アクティブパネル、インスペクター、モーダル | 100% |
| ミッドグラウンド | 1.2〜2.5m | ノードグラフ、接続線、ワークスペース | 100% |
| バックグラウンド | 2.5〜5.0m | オーバービューマップ、アンビエントステータス | 40〜70% |

### 3DのノードグラフNode

**データはユーザーに向かって流れます。** ノードは実行順序に沿って z 軸に配置されます：

```
USER (here)
  z=0.0m   [Output Nodes]     -- Results
  z=0.3m   [Transform Nodes]  -- Processors
  z=0.6m   [Agent Nodes]      -- LLM calls
  z=0.9m   [Retrieval Nodes]  -- RAG, APIs
  z=1.2m   [Input Nodes]      -- Triggers
```

並列ブランチは水平方向（x軸）に広がります。条件分岐ブランチは垂直方向（y軸）に広がります。

**ノードの表現（3 LOD レベル）：**
- **LOD-0**（静止、1.5m超）：タイプアイコン、名前、ステータスグローを持つ12×8cmのすりガラスの長方形
- **LOD-1**（ホバー、400msの視線）：14×10cmに拡大し、ポートと最終実行情報を表示
- **LOD-2**（選択済み）：フォアグラウンドにスライドし、ライブ設定編集付きの30×40cmの詳細パネルに拡大

**発光チューブとしての接続線：**
- 静止時の直径4mm、データを運ぶとき8mm
- データタイプ別の色分け（白=テキスト、シアン=構造化、マゼンタ=画像、アンバー=音声、緑=ツール呼び出し）
- フロー方向と速度を示すアニメーションパーティクル
- 同レイヤー間で3本以上並行する場合は自動バンドル

### 7つのエージェント状態

| 状態 | エッジグロー | 内部 | 音 | パーティクル |
|-------|-----------|----------|-------|-----------|
| アイドル | 定常緑、低輝度 | 静的なすりガラス | なし | なし |
| キュー中 | 点滅アンバー、1Hz | 微かな回転 | なし | 入力端でゆっくりドリフト |
| 実行中 | 定常ブルー、中輝度 | アニメーションシマー | ソフトな空間ハム | 接続線で急速なフロー |
| ストリーミング中 | ブルー + 出力ストリーム | シマー + テキスト断片 | ハム | テキスト断片が前方に流れる |
| 完了 | 白くフラッシュ後、緑 | 静的 | 完了チャイム | なし |
| エラー | 点滅赤、2Hz | 赤みがかった色調 | アラートトーン（1回） | なし |
| 一時停止 | 定常アンバー | フリーズフレーム + 一時停止アイコン | なし | 所定の位置で凍結 |

### コラボレーションプレゼンス

各コラボレーターは以下で表現されます：
- **ヘッドプロキシ：** 頭の向きに合わせて回転するプロフィール画像付きの半透明の球体

### トランジションコレオグラフィー

すべてのトランジションは道案内に役立ちます。主要なトランジションは最大600ms、副次的なものは200ms、選択は0ms。

| トランジション | 所要時間 | 主要な動き |
|-----------|----------|------------|
| オーバービューからフォーカスへ | 600ms | カメラがターゲットにドリフト、他の領域が30%にフェード |
| フォーカスから詳細へ | 500ms | ノードが前方にスライドし、拡大し、接続線がハイライト |
| 詳細からオーバービューへ | 600ms | パネルが折りたたまれ、ノードが後退し、全トポロジーが可視化 |
| ゾーン切り替え | 500ms | 現在がサイドにスライドアウト、新しいものがスライドイン |
| ウィンドウから没入型へ | 1000ms | ボーダーが溶解、ノードが完全な空間位置に拡大 |

### 快適さの対策

- ユーザーアクションなしのカメラ主導の動きなし
- 安定した水平線（水平面は傾かない）
- 主要なインタラクションは0.8〜2.5m、視線から±15度の範囲内
- 45分後に休憩プロンプト（モーダルではなくアンビエント照明の変化）
- 高速移動時の周辺ビネット
- 頻繁に使用するコントロールはすべて腕を体の横に下げた状態でアクセス可能（手首/指のみ）

## 10. エージェント横断シンセシス

### 8エージェント全員が同意した点

1. **2Dファースト、空間はセカンド。** すべてのエージェントが独立してこの結論に至りました。まず優れたウェブダッシュボードを構築し、その後徐々に空間機能を追加する。

2. **デバッグがキラーユースケース。** プロダクトリサーチャー、UXリサーチャー、XRインターフェースアーキテクトの全員が収束した：ワークフロー構造上のランタイムトレースの空間オーバーレイこそ、3Dが2Dを真に上回る部分。

3. **初期リーチには VisionOS より WebXR。** Vision Pro の約100万台のインストールベースではビジネスを維持できない。ブラウザの WebXR が流通の突破口。

4. **「ウォールーム」コラボレーションシナリオ。** 複数のエージェントが共同インシデント対応を最強の空間バリュープロポジションとして強調した ── チームが共有3D空間に入り、障害中のパイプラインを一緒にデバッグする。

5. **プログレッシブディスクロージャーが不可欠。** UXリサーチ、空間UI、サポートのすべてが、空間の複雑さは徐々に明らかにされるべきであり、初回ユーザーに一度に投入されるべきではないと強調した。

6. **音声がパワーユーザーのアクセラレーター。** UXリサーチャーとXRインターフェースアーキテクトの両方が、音声コマンドを「空間コンピューティングのコマンドライン」として特定した ── アクセシビリティとエキスパートの効率性に不可欠。

### 解決すべき主要な緊張関係

| 緊張 | 立場A | 立場B | 必要な解決 |
|---------|-----------|-----------|-------------------|
| **価格設定** | グロースハッカー：ユーザー月29〜59ドル | トレンドリサーチャー：ユーザー月99〜249ドル | ベータでのA/Bテスト |
| **VisionOS の優先度** | アーキテクチャ：フェーズ3（第13週以降） | 空間UI：仕様完成 | WebXR 優先、検証後にVisionOS |
| **オーケストレーション言語** | アーキテクチャ：Rust | プロジェクト計画：未指定 | Rust はパフォーマンスクリティカルな DAG 実行に正しい |
| **MVP スコープ** | アーキテクチャ：フェーズ1は2Dのみ | ブランド：空間でリード | 2Dファーストだが、すべてのデモで空間を確実に含める |
| **コミュニティプラットフォーム** | サポート：Discordファースト | マーケティング：Discord + オープンソース | 両方 ── コミュニティには Discord、開発者エンゲージメントには GitHub |

### この演習が示すもの

このディスカバリードキュメントは、共有された目標に各自の深いドメイン専門知識をもたらしながら並行して実行された8つの専門エージェントによって作成されました。エージェントたちは一貫した結論に独立して到達しつつ、単一のジェネラリストが生み出すことが難しいドメイン固有のインサイトを表面化させました：

- **プロダクトトレンドリサーチャー**は、戦略全体を再構築した Vision Pro の販売データを発見した
- **バックエンドアーキテクト**は、マーケティング重視のチームが考えなかったであろう Rust オーケストレーションエンジンを設計した
- **ブランドガーディアン**は、既存のカテゴリで競うのではなくカテゴリ（「SpatialAIOps」）を創造した
- **UXリサーチャー**は、空間コンピューティングがパラメータータスクに摩擦を生むという直感に反する発見をした
- **XRインターフェースアーキテクト**は、自然な空間認知にマッピングされる「データがあなたに向かって流れる」トポロジーを設計した
- **プロジェクトシェパード**は、タイムライン全体を頓挫させる可能性のある3つのクリティカルなボトルネック役割を特定した
- **グロースハッカー**は、空間コンピューティング固有のシェアしやすさに特化したバイラルループを設計した
- **サポートレスポンダー**は、製品自身のAI機能をサポートの差別化要因に変えた

その結果、AIエージェントのエージェンシーが協調して単一セッションで生み出した、実際の開発の基礎として機能しうる包括的なクロスファンクショナルな製品計画が完成しました。
