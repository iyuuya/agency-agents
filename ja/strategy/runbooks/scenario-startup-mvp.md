# 🚀 ランブック: スタートアップ MVP 構築

> **モード**: NEXUS-スプリント | **期間**: 4〜6週間 | **エージェント数**: 18〜22

---

## シナリオ

スタートアップの MVP を構築します — プロダクトマーケットフィットを素早く検証する必要がある新製品です。スピードが重要ですが、品質も同様です。4〜6週間でアイデアからリアルユーザーがいるライブ製品まで到達する必要があります。

## 週次実行計画

### 1週目: ディスカバリー + アーキテクチャ（フェーズ0 + フェーズ1 を圧縮）

```
1〜2日目: 圧縮ディスカバリー
├── Trend Researcher → クイック競合スキャン（1日、フルレポートではない）
├── UX Architect → 主要ユーザーフローのワイヤーフレーム
└── Senior Project Manager → 仕様からタスクリストへの変換

3〜4日目: アーキテクチャ
├── UX Architect → CSS デザインシステム + コンポーネントアーキテクチャ
├── Backend Architect → システムアーキテクチャ + データベーススキーマ
├── Brand Guardian → クイックブランド基盤（カラー、タイポグラフィ、ボイス）
└── Sprint Prioritizer → RICE スコア付きバックログ + スプリント計画

5日目: 基盤セットアップ
├── DevOps Automator → CI/CD パイプライン + 環境
├── Frontend Developer → プロジェクトスキャフォルディング
├── Backend Architect → データベース + API スキャフォールド
└── 品質ゲート: アーキテクチャパッケージの承認
```

### 2〜3週目: コア構築（フェーズ2 + フェーズ3）

```
スプリント1（2週目）:
├── Agents Orchestrator が Dev↔QA ループを管理
├── Frontend Developer → コア UI（認証、メインビュー、ナビゲーション）
├── Backend Architect → コア API（認証、CRUD、ビジネスロジック）
├── Evidence Collector → 完了した各タスクの QA
├── AI Engineer → 該当する場合は ML 機能
└── 週末にスプリントレビュー

スプリント2（3週目）:
├── 残りの機能に対して Dev↔QA ループを継続
├── Growth Hacker → バイラルメカニズム + リファラルシステムの設計
├── Content Creator → ローンチコンテンツの制作開始
├── Analytics Reporter → トラッキングとダッシュボードのセットアップ
└── 週末にスプリントレビュー
```

### 4週目: 磨き上げとハードニング（フェーズ4）

```
1〜2日目: 品質スプリント
├── Evidence Collector → 完全なスクリーンショットスイート
├── Performance Benchmarker → 負荷テスト
├── Frontend Developer → QA 問題の修正
├── Backend Architect → API 問題の修正
└── Brand Guardian → ブランド一貫性監査

3〜4日目: リアリティチェック
├── Reality Checker → 最終統合テスト
├── Infrastructure Maintainer → 本番稼働準備
└── DevOps Automator → 本番デプロイメント準備

5日目: ゲート判定
├── Reality Checker の判定
├── 要修正の場合: クイック修正サイクル（2〜3日）
├── 合格の場合: ローンチへ移行
└── Executive Summary Generator → ステークホルダーブリーフィング
```

### 5〜6週目: ローンチと成長（フェーズ5）

```
5週目: ローンチ
├── DevOps Automator → 本番デプロイメント
├── Growth Hacker → 獲得チャネルの起動
├── Content Creator → ローンチコンテンツの公開
├── Social Media Strategist → クロスプラットフォームキャンペーン
├── Analytics Reporter → リアルタイムモニタリング
└── Support Responder → ユーザーサポートの稼働

6週目: 最適化
├── Growth Hacker → チャネルの分析と最適化
├── Feedback Synthesizer → 初期ユーザーフィードバックの収集
├── Experiment Tracker → A/B テストの開始
├── Analytics Reporter → 1週目分析
└── Sprint Prioritizer → イテレーションスプリントの計画
```

## 重要な意思決定

| 意思決定ポイント | タイミング | 決定者 |
|---------------|------|-------------|
| コンセプトの Go/No-Go | 2日目終了時 | スタジオプロデューサー |
| アーキテクチャの承認 | 4日目終了時 | シニアプロジェクトマネージャー |
| MVP の機能スコープ | スプリントプランニング時 | スプリント優先度設定ツール |
| 本番稼働準備 | 4週目5日目 | リアリティチェッカー |
| ローンチタイミング | リアリティチェッカーの合格後 | スタジオプロデューサー |

## 成功基準

| メトリクス | 目標 |
|--------|--------|
| ライブ製品までの期間 | ≤ 6週間 |
| コア機能の完成度 | MVP スコープの100% |
| 初期ユーザーのオンボーディング | ローンチから 48 時間以内 |
| システム稼働率 | 最初の1週間 > 99% |
| 収集されたユーザーフィードバック | 最初の2週間で ≥ 50件の回答 |

## よくある落とし穴と対策

| 落とし穴 | 対策 |
|---------|-----------|
| 構築中のスコープクリープ | スプリント優先度設定ツールが MoSCoW を徹底 — 「しない」は本当にしない |
| スケールを見越した過剰エンジニアリング | ラピッドプロトタイパーのマインドセット — まず検証し、スケールは後で |
| スピードのために QA をスキップする | エビデンスコレクターはすべてのタスクに対して実行 — 例外なし |
| モニタリングなしのローンチ | インフラメンテナーが1週目にモニタリングをセットアップ |
| フィードバック機構がない | アナリティクス + フィードバック収集をスプリント1 に組み込み |
