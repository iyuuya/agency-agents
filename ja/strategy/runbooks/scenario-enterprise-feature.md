# 🏢 ランブック: エンタープライズ機能開発

> **モード**: NEXUS-スプリント | **期間**: 6〜12週間 | **エージェント数**: 20〜30

---

## シナリオ

既存のエンタープライズ製品に主要機能を追加します。コンプライアンス、セキュリティ、品質ゲートは交渉の余地がありません。複数のステークホルダーとの調整が必要です。機能は既存システムとシームレスに統合されなければなりません。

## 実行計画

### フェーズ1: 要件とアーキテクチャ（1〜2週目）

```
1週目: ステークホルダー調整
├── Project Shepherd → ステークホルダー分析 + コミュニケーション計画
├── UX Researcher → 機能ニーズに関するユーザーリサーチ
├── Legal Compliance Checker → コンプライアンス要件のスキャン
├── Senior Project Manager → 仕様からタスクへの変換
└── Finance Tracker → 予算フレームワーク

2週目: 技術アーキテクチャ
├── UX Architect → UX 基盤 + コンポーネントアーキテクチャ
├── Backend Architect → システムアーキテクチャ + 統合計画
├── UI Designer → コンポーネントデザイン + デザインシステムの更新
├── Sprint Prioritizer → RICE スコア付きバックログ
├── Brand Guardian → ブランドへの影響評価
└── 品質ゲート: アーキテクチャレビュー（Project Shepherd + Reality Checker）
```

### フェーズ2: 基盤（3週目）

```
├── DevOps Automator → フィーチャーブランチパイプライン + フィーチャーフラグ
├── Frontend Developer → コンポーネントスキャフォルディング
├── Backend Architect → API スキャフォールド + データベースマイグレーション
├── Infrastructure Maintainer → ステージング環境のセットアップ
└── 品質ゲート: 基盤の確認（Evidence Collector）
```

### フェーズ3: 構築（4〜9週目）

```
スプリント1〜3（4〜9週目）:
├── Agents Orchestrator → Dev↔QA ループの管理
├── Frontend Developer → UI 実装（タスクごと）
├── Backend Architect → API 実装（タスクごと）
├── Senior Developer → 複雑/プレミアム機能
├── Evidence Collector → 各タスクの QA（スクリーンショット）
├── API Tester → 各 API タスクのエンドポイント検証
├── Experiment Tracker → 主要機能の A/B テストセットアップ
│
├── 隔週:
│   ├── Project Shepherd → ステークホルダー状況更新
│   ├── Executive Summary Generator → エグゼクティブブリーフィング
│   └── Finance Tracker → 予算追跡
│
└── ステークホルダーデモを含むスプリントレビュー
```

### フェーズ4: ハードニング（10〜11週目）

```
10週目: エビデンス収集
├── Evidence Collector → 完全なスクリーンショットスイート
├── API Tester → 完全なリグレッションスイート
├── Performance Benchmarker → 10倍トラフィックでの負荷テスト
├── Legal Compliance Checker → 最終コンプライアンス監査
├── Test Results Analyzer → 品質メトリクスダッシュボード
└── Infrastructure Maintainer → 本番稼働準備

11週目: 最終判定
├── Reality Checker → 統合テスト（デフォルト: 要修正）
├── 必要に応じた修正サイクル（2〜3日）
├── 再検証
└── Executive Summary Generator → Go/No-Go 推奨事項
```

### フェーズ5: ロールアウト（12週目）

```
├── DevOps Automator → カナリアデプロイメント（5% → 25% → 100%）
├── Infrastructure Maintainer → リアルタイムモニタリング
├── Analytics Reporter → 機能採用のトラッキング
├── Support Responder → 新機能のユーザーサポート
├── Feedback Synthesizer → 早期フィードバックの収集
└── Executive Summary Generator → ローンチレポート
```

## 品質要件

| 要件 | しきい値 | 検証方法 |
|-------------|-----------|-------------|
| コードカバレッジ | > 80% | テスト結果アナライザー |
| API 応答時間 | P95 < 200ms | パフォーマンスベンチマーカー |
| アクセシビリティ | WCAG 2.1 AA | エビデンスコレクター |
| セキュリティ | クリティカル脆弱性ゼロ | 法的コンプライアンスチェッカー |
| ブランド一貫性 | 95%以上の準拠 | ブランドガーディアン |
| 仕様準拠 | 100% | リアリティチェッカー |
| 負荷対応 | 現在トラフィックの10倍 | パフォーマンスベンチマーカー |

## リスク管理

| リスク | 発生確率 | 影響度 | 軽減策 | オーナー |
|------|------------|--------|-----------|-------|
| 統合の複雑さ | 高 | 高 | 早期統合テスト、各スプリントでの API テスター活用 | バックエンドアーキテクト |
| スコープクリープ | 中 | 高 | スプリント優先度設定ツールが MoSCoW を適用、プロジェクトシェパードが変更管理 | スプリント優先度設定ツール |
| コンプライアンス問題 | 中 | クリティカル | 法的コンプライアンスチェッカーを初日から関与 | 法的コンプライアンスチェッカー |
| パフォーマンスの低下 | 中 | 高 | 毎スプリントのパフォーマンスベンチマーカーによるテスト | パフォーマンスベンチマーカー |
| ステークホルダーの方向性のズレ | 低 | 高 | 隔週のエグゼクティブブリーフィング、プロジェクトシェパードの調整 | プロジェクトシェパード |
